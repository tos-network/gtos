# GTOS vs geth Deep Audit — Round 2 (2026-03-08)

Baseline: go-ethereum v1.10.25 (~/geth) vs GTOS (~/gtos)

Scope: subsystems not fully covered in Round 1 (BUGS-20260308.md) — consensus engine
internals, tx pool, RPC/API, P2P sync, cryptographic paths, LVM/state transition,
core state management, genesis handling.

Each issue independently verified against source code. False positives removed.

---

## Severity Levels

- **HIGH**: Active exploit vector, DoS, or consensus-breaking bug.
- **MEDIUM**: Correctness gap, compatibility break, or defense-in-depth issue.
- **LOW**: Minor issue, informational, or cosmetic.

## HIGH

### H-1: Gas limit verification uses canonical parent during reorgs

**File**: `consensus/dpos/dpos.go` — `verifyHeader()`, line 1150

**Problem**: The gas limit rate-of-change check uses `chain.GetHeaderByNumber(number - 1)`
which fetches the canonical-chain parent, not the actual parent linked by `ParentHash`.
During a reorg, this checks gas limit delta against the wrong parent. Additionally,
when the parent is `nil`, the check is silently skipped (`if parent != nil`) rather
than returning `ErrUnknownAncestor`.

The correct parent lookup (by hash) happens later in `verifyCascadingFields()` at
line 1173, but that function only checks `GasUsed <= GasLimit`, not the inter-block
gas limit delta. So the `VerifyGaslimit` call in `verifyHeader` is the sole delta
check and it has both bugs.

**Impact**: During reorgs, an attacker could craft blocks with arbitrarily large gas
limit jumps that pass verification. Nodes with different header availability states
could disagree on block validity.

**Verified**: CONFIRMED — line 1150 uses `GetHeaderByNumber`, not `GetHeader(hash)`.
`verifyCascadingFields` at line 1178 only checks `GasUsed <= GasLimit`.

**Status**: OPEN

### H-2: TX pool missing pre-lock signature verification (DoS)

**File**: `core/tx_pool.go` — `addTxs()`, lines 955-973

**Problem**: geth's `addTxs` (lines 889-913) calls `types.Sender(pool.signer, tx)`
**before** acquiring `pool.mu.Lock()`, rejecting invalid signatures cheaply without
holding the lock. gtos removed this pre-lock check entirely. All signature verification
happens inside `addTxsLocked` → `add()` → `ResolveSender()` under the global mutex.

**Impact**: An attacker can flood the pool with transactions bearing invalid signatures.
Each one acquires the global mutex before signature is checked, serializing all pool
operations behind expensive crypto verification. Degrades throughput for legitimate
transactions.

**Verified**: CONFIRMED — geth has pre-lock `types.Sender` at line 902; gtos `addTxs`
goes directly to `news = append(news, tx)` with no signature check before the lock.

**Status**: OPEN

---

### H-3: UnoDecryptBalance CPU/memory DoS via unbounded maxAmount

**File**: `internal/tosapi/api.go` — `UnoDecryptBalance()`, lines 3176-3179

**Problem**: `UnoDecryptBalance` accepts a user-supplied `maxAmount` parameter
(`*hexutil.Uint64`) with no upper-bound cap. `SolveDiscreteLog` in
`crypto/uno/ecdlp.go:25` uses Baby-Step Giant-Step with `O(sqrt(maxAmount))` time
and space. A caller can pass `maxAmount = 2^64 - 1`, which attempts to allocate a
hash map with ~2^32 entries (~160 GB RAM) and perform ~2^32 elliptic curve operations.

Default of 1,000,000,000 (when nil) is safe, but the user can override it.

**Impact**: A single RPC call can OOM-crash the node or tie up a CPU core indefinitely.

**Verified**: CONFIRMED — line 3178 `effectiveMax = uint64(*maxAmount)` with no cap.
`SolveDiscreteLog` allocates `make(map[string]uint64, n+1)` where `n = ceil(sqrt(max))`.

**Status**: OPEN

### H-4: nocgo builds reject all UNO proofs — consensus partition risk

**File**: `crypto/ed25519/uno_proofs_nocgo.go`

**Problem**: In nocgo builds (without CGO or `ed25519c` build tag), all 28 UNO proof
verification/generation functions unconditionally return `ErrUNOBackendUnavailable`.
If confidential transactions are enabled on a chain, nocgo-built nodes cannot validate
blocks containing them.

**Impact**: Consensus partition between cgo and nocgo builds when UNO transactions
are present.

**Verified**: CONFIRMED — build constraint `//go:build !cgo || !ed25519c`; every
function returns `ErrUNOBackendUnavailable`. This is BY DESIGN for build compatibility,
but no startup-time guard warns operators.

**Status**: OPEN (needs startup check or documentation)

## MEDIUM

### M-1: LVM.Call non-revert errors don't consume all gas

**File**: `core/vm/lvm.go` — `Call()`, lines 170-179

**Problem**: In geth's `EVM.Call` (evm.go:235-239), non-revert errors set `gas = 0`,
consuming all remaining gas. In LVM.Call, on any error (including non-revert), only
actually-used gas is consumed — leftover gas is returned to the caller:

```go
return nil, gas - consumed, execErr  // leftover returned
```

**Impact**: Failed LVM calls are cheaper than equivalent failed EVM calls. Changes the
economics of contract interactions and could be exploited in griefing scenarios.

**Verified**: CONFIRMED — geth sets `gas = 0` for non-revert errors; LVM returns
`gas - consumed`.

**Status**: OPEN

---

### M-2: No snapshot/revert for sysaction/slash/UNO error paths

**File**: `core/state_transition.go` — lines 372-400

**Problem**: `sysaction.Execute()`, `slashindicator.Execute()`, and `applyUNO()` operate
directly on the StateDB with no protective snapshot. If a handler partially writes to
state before returning an error, those partial writes persist. Unlike LVM.Call (which
handles snapshot/revert internally), these native execution paths have no equivalent
protection.

**Impact**: Partial state corruption on failed system actions or confidential
transactions.

**Verified**: CONFIRMED — no `st.state.Snapshot()` before any of the three execution
paths. Sysaction handlers receive the real StateDB and can write before erroring.

**Status**: OPEN

---

### M-3: tos.staticcall skips snapshot/revert (defense-in-depth gap)

**File**: `core/vm/lvm.go` — `tos.staticcall`, lines 2858-2932

**Problem**: `tos.staticcall` takes no snapshot and performs no `RevertToSnapshot` on
error. The code comments: "No state was written (readonly), so no snapshot revert
needed." By contrast, `tos.call` (line 2775) takes a snapshot and reverts on error.
geth's `StaticCall` also takes a snapshot (evm.go:351) even for read-only calls.

**Impact**: If any future host function added without a readonly guard is called from
staticcall, state mutations would silently persist. Defense-in-depth gap.

**Verified**: CONFIRMED (BY DESIGN) — explicit comment documents the decision. No
current state mutation possible, but no defensive protection either.

**Status**: OPEN

### M-4: secp256r1 signature malleability — no low-S enforcement

**File**: `accountsigner/crypto.go` — `VerifyRawSignature`, lines 636-643

**Problem**: For secp256k1, `ValidateSignatureValues` enforces `s <= secp256k1halfN`.
For secp256r1, `VerifyRawSignature` calls `ecdsa.Verify()` directly with no low-S
check. Given a valid signature `(r, s)`, the alternate `(r, N-s)` is equally valid.

**Impact**: Transaction malleability for secp256r1-signed transactions. An observer
can flip S to produce a different valid tx hash for the same logical transaction.

**Verified**: CONFIRMED — no S normalization check exists for secp256r1 anywhere in
the codebase.

**Status**: OPEN

### M-5: ed25519 signature malleability — cgo/nocgo divergence risk

**Files**: `crypto/ed25519/ed25519_nocgo.go:22`, `crypto/ed25519/ed25519_cgo.go:208`

**Problem**: The nocgo path delegates to Go's `crypto/ed25519.Verify`, which may or
may not enforce `S < L` depending on Go version (Go 1.21+ does, earlier does not).
The cgo path calls `gtos_ed25519_verify` — whether the C library checks `S < L`
depends on the library. Neither Go wrapper adds an explicit S-range check.

**Impact**: cgo and nocgo builds may produce different accept/reject results for the
same malleable signature, creating a consensus divergence between build configurations.

**Verified**: CONFIRMED — neither wrapper adds explicit S-range validation. Behavior
depends on underlying library versions.

**Status**: OPEN

### M-6: Receipt field `effectiveTxPrice` vs standard `effectiveGasPrice`

**File**: `internal/tosapi/api.go` — lines 1662-1665

**Problem**: gtos returns `effectiveTxPrice` in receipt JSON. The Ethereum JSON-RPC
standard uses `effectiveGasPrice`. Any tooling, wallet, or block explorer reading
`effectiveGasPrice` will get `undefined`.

**Verified**: CONFIRMED — field name is `effectiveTxPrice` in gtos vs
`effectiveGasPrice` in geth.

**Status**: OPEN

### M-7: RPCTransaction struct missing `gasPrice` field

**File**: `internal/tosapi/api.go` — `RPCTransaction` struct, lines 1314-1334

**Problem**: geth includes `GasPrice *hexutil.Big \`json:"gasPrice"\`` in
`RPCTransaction`. gtos removed this field entirely. Every transaction returned via
`tos_getTransactionByHash`, `tos_getBlockByNumber(fullTx=true)`, etc. will be missing

**Verified**: CONFIRMED — no `gasPrice` field in the struct or `newRPCTransaction`.

**Status**: OPEN

### M-8: newRevertError swallows human-readable reason

**File**: `internal/tosapi/api.go` — `newRevertError()`, lines 1064-1073

**Problem**: geth returns `"execution reverted: <decoded reason>"` in the error message
and raw hex in `ErrorData()`. gtos returns the generic `"execution reverted"` in the
error message and puts the decoded string (not raw hex) in `ErrorData()`.

Two breaks: (1) error message loses the human-readable reason; (2) `ErrorData()`
returns decoded text where clients expect raw hex ABI data.

**Verified**: CONFIRMED — gtos `error: errors.New("execution reverted")` always;
`reason: decoded` string vs geth's `reason: hexutil.Encode(result.Revert())`.

**Status**: OPEN

### M-9: Checkpoint vote gossip has no sqrt(N) fan-out

**File**: `tos/handler.go` — lines 581-607

**Problem**: `BroadcastCheckpointVote` and `RelayCheckpointVote` send to ALL connected
peers. Block broadcast at line 631 uses `sqrt(N)` fan-out. The LRU seen-set (added in
Round 1 fix) prevents infinite loops but does not limit the O(N) per-hop fan-out.

**Impact**: With V validators and N peers per node, each checkpoint generates V*N
messages per node. Bandwidth waste during normal operation.

**Verified**: CONFIRMED — `BroadcastCheckpointVote` iterates `h.peers.allPeers()`;
`BroadcastBlock` uses `peers[:int(math.Sqrt(float64(len(peers))))]`.

**Status**: OPEN

### M-10: No per-peer `knownVotes` tracking on Peer

**File**: `tos/protocols/tos/peer.go` — lines 52-78, 287-290

**Problem**: The `Peer` struct tracks `knownBlocks` and `knownTxs` caches to avoid
duplicate sends. There is no `knownVotes` cache. `SendCheckpointVote` sends directly
without checking if the peer already has the vote.

**Impact**: If peer A and peer B both send the same vote to node X, node X relays
back to B (since excludeID=A). Produces O(N^2) redundant messages across the network.

**Verified**: CONFIRMED — no `knownVotes` field in Peer struct; `SendCheckpointVote`
has no dedup check.

**Status**: OPEN

### M-11: Scheduled task gas produces no receipt

**File**: `core/state_processor.go` — lines 88-104

**Problem**: `RunScheduledTasks` runs before `ExecuteTransactions`. Its gas is added to
`usedGas` and adjusted in existing receipt `CumulativeGasUsed` fields, but no receipt
is generated for the scheduled tasks themselves. If a block has scheduled tasks but
zero user transactions, the gas is consumed with no receipt at all.

**Verified**: CONFIRMED — `RunScheduledTasks` at line 88 returns gas but no receipt
is created. Line 101 adjusts existing receipts only.

**Status**: OPEN

Testnet genesis state is identical to mainnet's initial allocation.

### M-13: Missing EIP-1559 fee cap preCheck validation

**File**: `core/state_transition.go` — `preCheck()`, lines 243-263

**Problem**: geth's `preCheck()` validates `gasFeeCap >= gasTipCap`, `gasFeeCap >= baseFee`,
and bit-length bounds. gtos removed all London/EIP-1559 checks. While gtos uses a
fixed-price model, the `gasFeeCap`/`gasTipCap` fields still exist and affect
`buyGas()` balance calculations.

**Verified**: CONFIRMED (BY DESIGN) — consistent with fixed-price model, but checks
are absent.

**Status**: BY DESIGN

### M-14: `encodeSigHeader` silently drops RLP encode errors

**File**: `consensus/dpos/dpos.go` — line 630

**Problem**: `rlp.Encode(w, enc)` return value (error) is silently discarded. Clique
panics on encode failure (clique.go:746-748). Silent failure could produce incorrect
seal hashes.

**Verified**: CONFIRMED — `rlp.Encode(w, enc)` with no error check. Clique does
`if err := rlp.Encode(...); err != nil { panic(...) }`.

## LOW

### L-1: Block header Nonce field not validated as zero

**File**: `consensus/dpos/dpos.go` — `verifyHeader()`

`Prepare()` sets `header.Nonce = BlockNonce{}` but `verifyHeader()` never validates
that Nonce is zero. Non-zero Nonce passes verification, creating an 8-byte free
data channel in block headers. Nonce is covered by the block seal, so no forgery risk.

**Verified**: CONFIRMED

---

### L-2: Missing `VerifyForkHashes` call

**File**: `consensus/dpos/dpos.go` — `verifyHeader()`

Clique calls `misc.VerifyForkHashes(chain.Config(), header, false)`. DPoS does not.
The function exists in the gtos codebase (`consensus/misc/forks.go`) but is never
called. Future fork hash protections would be silently bypassed.

**Verified**: CONFIRMED

---

### L-3: Missing `FullImmutabilityThreshold` snapshot fallback

**File**: `consensus/dpos/dpos.go` — `snapshot()`

Clique's `snapshot()` has a fallback when headers walked exceeds
`FullImmutabilityThreshold` or parent is missing (light client). DPoS `snapshot()`
has no such fallback. The backward header walk could accumulate an unbounded number
of headers in memory.

**Verified**: CONFIRMED

---

### L-4: `CallCtx.Depth` hardcoded to 0 in `LVM.Call`

**File**: `core/vm/lvm.go` — line 165

`LVM.Call` sets `Depth: 0` in `CallCtx` regardless of `l.depth`. Internal nested
calls correctly propagate `ctx.Depth + 1`, but the top-level call always starts at 0.
If `LVM.Call` is invoked recursively, the per-call-level `maxCallDepth = 8` limit
can be exceeded.

**Verified**: CONFIRMED — line 165 hardcodes `Depth: 0` despite `l.depth` tracking.

---

### L-5: Missing `CreateAccount` for calls to non-existing addresses

**File**: `core/vm/lvm.go` — `Call()`, lines 146-186

geth's `EVM.Call` explicitly calls `CreateAccount` before Transfer (evm.go:194).
LVM.Call does not. `AddBalance` in Transfer implicitly creates the account in most
StateDB implementations, but the explicit creation step and EIP-158 zero-value
short-circuit are both missing.

**Verified**: CONFIRMED — no `CreateAccount` call in LVM.Call.

---

### L-6: `normalizeChainID` accepts zero chainID

**File**: `core/types/transaction_signing.go` — lines 82-87

`normalizeChainID` converts `nil` to `new(big.Int)` (zero). If the signer's chainId
is also zero (from nil config), zero-chainID transactions would be accepted, weakening
EIP-155 replay protection. In practice, production nodes always have non-zero chain ID.

**Verified**: CONFIRMED — nil → zero conversion; zero-chainID tx accepted if signer
also has zero chainID.

---

### L-7: `DeveloperGenesisBlock` pre-funds non-existent EVM precompile addresses

**File**: `core/genesis.go` — lines 520-529

Pre-funds addresses 1-9 (ECRecover, SHA256, etc.) with 1 wei. GTOS uses the LVM,
not EVM — these precompile contracts don't exist. Dead weight from geth copy.

**Verified**: CONFIRMED — addresses 1-9 pre-funded; LVM has no EVM precompiles.

---

### L-8: `GasPrice` API returns fixed value with no error return

**File**: `internal/tosapi/api.go` — line 51

geth returns `(*hexutil.Big, error)`; gtos returns `*hexutil.Big` (no error).
Intentional simplification for fixed-price model but diverges from Ethereum JSON-RPC
interface.

**Verified**: BY DESIGN
