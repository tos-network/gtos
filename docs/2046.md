# GTOS 2046 Architecture: OpenFox as Intent and Terminals, TOL as the Agent-Native Contract Layer, and GTOS as Settlement, Policy, and Agent Infrastructure

## Overview

This document defines the architecture split that should guide the stack now if
the long-term goal is an AGI-native financial system by the year 2046.

The central thesis is:

- `OpenFox` should become the intent runtime and multi-terminal access layer.
- `TOL` should become the agent-native contract, policy, and automation
  language layer.
- `GTOS` should become the settlement, policy-enforcement, and
  agent-infrastructure layer.

This split matters because the future financial interface is unlikely to be one
wallet app. It will be a network of weak entry points:

- cards
- POS devices
- voice nodes
- public kiosks
- robots
- vehicles
- wearables

Those surfaces should not be treated as the wallet itself. They should be
treated as temporary entry points into a persistent personal agent, a
machine-readable contract layer, and a policy-controlled on-chain account
system.

The core redefinition is this:

> The wallet in the 2046 model is not an app, a card, a kiosk, or a device.
> It is a policy-bound programmable account coordinated by OpenFox,
> described by TOL, and enforced by GTOS.

In short:

> OpenFox should own intent and terminals.  
> TOL should own machine-readable contract semantics.  
> GTOS should own settlement and authority.

## Non-Goals

The 2046 architecture should not pretend to solve every adjacent problem.
At this stage, it is not trying to be:

- a full cross-chain interoperability framework
- a complete fiat on-ramp and off-ramp compliance stack
- a universal offline-settlement protocol for arbitrary terminals
- a one-document specification for every oracle, marketplace, or consumer
  finance product that may later run on the stack

Those concerns may matter later, but they should not be allowed to blur the
core architecture:

- OpenFox as intent runtime and terminal layer
- TOL as contract-semantic layer
- GTOS as settlement, policy, and agent-infrastructure substrate

## Why This Must Start Now

The future architecture will not emerge by adding a better wallet UI later.
It must be designed into the system early.

If OpenFox is built as "just a frontend", if TOL is treated as "just a
compiler", and if GTOS is built as "just a payment chain", the system will
inherit the wrong abstractions:

- transaction-first UX instead of intent-first UX
- app-first access instead of terminal-first access
- opaque contract semantics instead of machine-readable automation
- off-chain policy logic instead of protocol-enforced policy
- hidden automation instead of auditable delegation
- single-device wallet assumptions instead of multi-surface authority

The correct move is to lay six technical foundations now:

1. intent-first transaction model
2. account contracts and policy engine
3. sponsor and gas abstraction
4. agent identity and discovery
5. audit logs and agent proofs
6. multi-terminal weak-entry design

## System Split

### OpenFox: Intent and Terminals

OpenFox should be the user-facing and agent-facing runtime that:

- ✅ captures and normalizes user intent
- ✅ runs the persistent personal agent
- ✅ manages user sessions across many terminal types
- ✅ explains approvals and outcomes in human terms
- ✅ selects providers, sponsors, and counterparties
- ✅ orchestrates requests across agent networks
- ✅ keeps a local operator and owner audit journal
- ✅ exposes terminal adapters for cards, POS, voice, kiosks, and robots

OpenFox should not be reduced to a browser wallet or a wallet settings panel.
It should be treated as the intent layer and terminal operating layer of the
agent-native financial stack.

### TOL: Agent-Native Contract and Policy Language

TOL should be the machine-readable contract layer that:

- ✅ gives agents first-class contract primitives for identity, capability,
  tasks, oracle slots, delegation, payment, and manifests
- ✅ exposes compiler-checked effects, gas bounds, and capability requirements
  as contract metadata
- ✅ compiles policy, automation, and financial coordination into artifacts that
  OpenFox can inspect and GTOS can execute
- ✅ should become the canonical authoring surface for agent-native policies,
  task flows, recovery logic, sponsor hooks, and verifiable automation
- ~ should define contract-level semantics that are legible to both humans and
  agents before execution

TOL should not be treated as only a compiler hidden behind GTOS.
It should be treated as the contract-semantic layer that lets the rest of the
stack speak machine-verifiable finance.

### GTOS: Settlement, Policy, and Agent Infrastructure

GTOS should be the backend financial fabric that:

- ✅ settles payments and value transfer
- ✅ hosts policy-controlled account contracts
- ✅ supports sponsor-backed execution and gas abstraction
- ✅ provides agent identity, capability, and reputation primitives
- ✅ provides discovery and gateway infrastructure
- ✅ provides verifiable receipts, settlement anchors, and callback traces
- ✅ supports scheduled actions, delegated authority, and recovery semantics
- ✅ supports public and privacy-tier asset paths

GTOS should not be positioned as a generic chain behind a wallet app.
It should be positioned as the trust and execution substrate for agent-native
finance.

## Core Design Principles

### 1. Intent First, Transaction Second

Users and terminals should speak in intent, not transaction internals.

OpenFox should model:

- what the user wants
- what limits apply
- what trust level is required
- what approval is needed
- what the acceptable execution envelope is

TOL should define contract semantics and metadata that make those actions
inspectable before execution.

GTOS should execute the resulting authority-checked action.

### 2. Terminal Is a Weak Surface

A card reader, robot, speaker, or public kiosk should not be a permanent root
of trust.

Each terminal should be treated as a weak surface with bounded authority.
The durable authority should live in:

- the personal agent
- the policy-bound account contract
- the signer mesh
- the protocol-level policy rules

### 3. Policy Must Be On-Chain or Policy-Bound

Limits, whitelists, delay windows, guardian approval, and recovery should not
live only in frontend code or server business logic.

The authoritative policy boundary must be enforced by GTOS account primitives.
The policy language and reusable machine-readable contract semantics should be
expressed through TOL artifacts where possible.

### 4. Sponsorship Must Be Native

Mass-market users should not manage gas directly.

The system should assume:

- sponsor discovery
- sponsor quotes
- bounded sponsor authorization
- gasless end-user UX

### 5. Agents Must Be Discoverable and Comparable

If finance becomes agent-mediated, users and personal agents must be able to:

- discover providers
- compare policy and price
- evaluate trust and reputation
- choose routes and counterparties dynamically

This cannot depend on hidden centralized provider registries forever.

### 6. Auditable Delegation Matters More Than Fancy UI

As more actions happen through automation and ambient interfaces, the core
trust question becomes:

who acted, under what policy, with what proof, through what contract semantics,
and with what settlement result?

That means receipts, proofs, logs, and machine-readable contract metadata
become strategic product features, not secondary operator details.

## Wallet Redefined

The 2046 model should explicitly reject the legacy wallet assumption.

A wallet is not:

- a downloadable mobile app
- a browser extension
- a key container tied to one device
- a branded screen for signing raw transactions

A wallet should instead be defined as:

- a policy-bound programmable account
- a persistent personal-agent context
- a portable authority model across weak terminals
- an auditable delegation and recovery system
- a settlement-linked identity and proof surface

OpenFox, TOL, and GTOS should therefore be designed around wallet continuity,
not around app continuity.

## Present-Day Interpretation of the 2046 Model

The 2046 design should be read as a near-term build direction, not only as a
futurist essay.

Today, the same split already makes sense:

- OpenFox is already a local-first runtime with wallet, policy, discovery,
  gateway, signer-provider, and paymaster-provider surfaces.
- TOL already exists as a real agent-native language and artifact pipeline for
  contract authoring, ABI metadata, package output, and deterministic execution
  on GTOS.
- GTOS already has or is specifying the right backend direction: sponsored
  execution, agent discovery, gateway, account abstraction direction, scheduled
  tasks, naming and registry systems, and privacy-tier asset support.

The next step is to stop treating these as separate features and start treating
them as one architecture.

## Four-Layer Action Model

To keep an intent-native system from collapsing back into a transaction-first
system, the stack should distinguish four different objects.

### 1. Intent

The desired outcome expressed by a human, terminal, or agent.

Examples:

- pay this merchant
- top up this service
- renew this subscription if price stays below a threshold
- move value privately to a known counterparty

### 2. Plan

A runtime-generated execution path that can satisfy the intent.
A plan may include routes, providers, sponsor choice, contracts, fallback
logic, and terminal-specific constraints.

A plan is not the same thing as a TOL artifact.
Instead, a plan should be allowed to reference one or more TOL artifacts and
their machine-readable semantics.

In the current stack, this object should primarily live in OpenFox and the
shared schema layer, because provider routing, sponsor selection, quote choice,
and retry behavior are runtime decisions.

### 3. Authorization

The approval and policy-binding object that binds a plan to allowed authority.
This layer determines who approved the action, under which limits,
through which terminal class, at what trust tier, and with which proof.

OpenFox should own the human-readable approval narrative and local approval
record.
GTOS should anchor the execution authorization boundary through account policy,
signer validation, and sponsor authorization.
TOL should make contract-side delegation, effects, and verifiability legible
before execution.

### 4. Receipt

The settlement-linked result preserved independently of the UI.
This layer captures what actually happened, who paid, which artifact was used,
what policy decision applied, and what proof references exist.

OpenFox should primarily own the Intent and Plan lifecycle.
TOL should make the contract semantics used by Plans and Authorizations
machine-readable.
GTOS should anchor Authorization and Receipt.

## Agent Identity Model

The word "agent" should be used precisely.
A durable architecture should distinguish the following:

### Agent Entity

The long-lived identity of an agent, whether personal, merchant-side,
solver-side, sponsor-side, or service-side.

### Agent Role

The role played in a given action, such as:

- requester
- actor
- provider
- sponsor
- signer
- gateway
- oracle
- counterparty
- guardian

### Agent Capability

The bounded set of functions or claims the agent is allowed to perform,
advertise, or verify.

### Agent Endpoint

The reachable interface through which the agent can be called, quoted,
discovered, or audited.

### Agent Account Binding

The relationship between the agent identity and the on-chain account,
policy scope, or settlement surface it may act upon.

This distinction matters because the same entity may play multiple roles,
and one account may be operated by multiple bounded agents under different
policies.

## Six Technical Pillars

### 1. Intent-First Transaction Model

OpenFox responsibilities:

- define a canonical intent schema
- normalize terminal input into intent objects
- maintain intent lifecycle state
- present human-readable approval prompts
- map one intent into zero, one, or many execution paths

TOL responsibilities:

- define contract-side machine-readable metadata for intent-relevant actions
- expose ABI, effects, gas, capability, delegation, and manifest fields in a
  stable way
- give OpenFox enough information to explain what a contract can do before it
  is called
- give GTOS deterministic artifacts to execute and verify

GTOS responsibilities:

- bind execution to a policy-aware envelope
- support intent-linked authorization metadata
- expose verifiable execution receipts
- preserve the final settlement trace independently of UI

Shared contract:

- ✅ `intent_id`
- ✅ `plan_id`
- ✅ `account_id`
- ✅ `terminal_class`
- ✅ `terminal_trust_tier`
- ✅ `actor_agent_id`
- ✅ `artifact_ref`
- ✅ `abi_ref`
- ✅ `policy_hash`
- ✅ `sponsor_policy_hash`
- ✅ `effects_hash`
- ✅ `approval_proof_ref`
- ✅ `receipt_ref`

The key point is that transaction schema alone is not enough.
There must be an intent schema above it and an inspectable contract schema
beside it.

### 2. Account Contract and Policy Engine

OpenFox responsibilities:

- policy authoring UX
- policy simulation and explanation
- policy class presets by terminal type
- risk summaries for operators and owners

TOL responsibilities:

- define account, policy, guardian, and recovery contract templates
- emit machine-readable policy metadata suitable for OpenFox inspection
- standardize reusable contract patterns for sponsor-aware and terminal-aware
  authorization

GTOS responsibilities:

- account contract validation
- spend limits
- whitelist and allowlist primitives
- delayed execution
- guardian and recovery flows
- terminal-scoped authority
- delegated agent authority
- revocation and suspension paths

This is the center of the architecture.
Without account contracts and policy enforcement, future multi-terminal finance
becomes unsafe by design.

### 3. Sponsor and Gas Abstraction

OpenFox responsibilities:

- sponsor discovery and ranking
- sponsor quote comparison
- sponsor fallback logic
- user-facing "total cost" UX instead of raw gas UX
- policy-aware sponsor selection

TOL responsibilities:

- define sponsor-aware authorization and delegated-call patterns at the contract
  level
- expose fee, value, and verifiability semantics in artifact metadata

GTOS responsibilities:

- one native transaction family with optional sponsor authorization
- sponsor nonce and expiry handling
- sponsor policy binding
- sender versus sponsor responsibility clarity
- clean receipt semantics for sponsored execution

This is the foundation for mass-market usability.

### 4. Agent Identity and Discovery

OpenFox responsibilities:

- provider discovery client flows
- quote and trust comparison
- route selection logic
- capability-aware UX and orchestration
- gateway session selection for private providers

TOL responsibilities:

- define capability, task, oracle, and agent-facing contracts as first-class
  reusable primitives
- emit manifests and capability requirements that OpenFox can compare and route
  against

GTOS responsibilities:

- agent registry
- capability registry
- reputation primitives
- discovery profile support
- gateway capability support
- settlement identity binding

If OpenFox is the intent runtime, it needs a marketplace of callable agents.
GTOS must provide the trust and discovery substrate for that marketplace.

### 5. Audit Logs and Agent Proofs

OpenFox responsibilities:

- local intent journal
- approval narrative
- owner-readable action history
- operator diagnostics
- replayable execution timeline

TOL responsibilities:

- emit machine-readable contract manifests and execution-relevant metadata that
  can be attached to proofs and receipts
- provide verifiable call semantics for agent-facing contract methods

GTOS responsibilities:

- signed receipts
- settlement anchors
- callback traces
- policy-decision references
- sponsor and signer attribution
- proof references for disputes and audits

Future agent-native finance will be judged less by how pretty its UI looks and
more by how defensible its machine actions are.

### 6. Multi-Terminal Weak-Entry Design

OpenFox responsibilities:

- terminal abstraction layer
- terminal-specific session handling
- terminal-specific confirmation patterns
- terminal-class risk model
- terminal adapter SDKs

Terminal classes should include from day one:

- cards
- POS devices
- voice nodes
- public terminals
- robot interfaces

TOL responsibilities:

- provide policy templates and contract patterns for terminal-class restricted
  authority
- make delegated and verifiable methods explicit so weak terminals can request
  bounded operations safely

GTOS responsibilities:

- terminal-bound permissions in account policy
- low-trust terminal caps
- high-trust terminal escalation rules
- recovery and revocation for lost terminal classes
- session and authorization proof hooks

The wallet must not be equated with one downloadable app.

### GTOS Owns

- settlement and balances

## Metadata Ownership and Consumption

The stack should be explicit about who owns which schema boundary.

Shared boundary package:

- owns the cross-repository field names, versions, and compatibility rules for
  `IntentEnvelope`, `PlanRecord`, `ApprovalRecord`, `ExecutionReceipt`,
  `artifact_ref`, `proof_ref`, and related shared objects

TOL:

- owns the contract-native metadata payload emitted by artifacts
- defines how effects, capability requirements, delegation, verifiability,
  recovery semantics, and manifest fields are represented inside TOL artifacts

OpenFox:

- should consume TOL metadata directly from artifacts or artifact-derived local
  records when rendering approvals, plans, and audits
- should not depend on GTOS RPC as the only way to understand contract
  semantics

GTOS:

- should preserve artifact and receipt references in execution-facing surfaces
- should not become the mandatory semantic translator for contract meaning

## Immediate Architecture Goal

The immediate architecture goal is not "build a futuristic terminal."
It is:

build the software and protocol seams that make future terminals possible
without redesigning the stack later.

That means:

- OpenFox should stop assuming that the primary client is an app or a browser.
- TOL should stop being treated as a hidden compiler detail and instead be
  treated as the contract-semantic layer for agent-native finance.
- GTOS should stop assuming that a raw transaction is the main user-facing
  object.

## Liveness, Availability, and Concurrency

Authority and audit are not enough.
The architecture also needs explicit behavior for degraded conditions.

### Weak-terminal availability

Weak terminals should be allowed to do only bounded work when connectivity or
the OpenFox runtime is degraded.

- fully offline arbitrary settlement is not an initial goal
- bounded store-and-forward or pre-authorized low-risk flows may become a
  future terminal class
- until then, weak terminals should fail closed or escalate, not silently
  exceed policy

### Intent timeout and fallback

Every intent and plan should have explicit timeout and expiry semantics.

- plans should expire independently of intents
- stale plans should be invalidated when policy, sponsor availability, or trust
  context changes
- provider failure should return control to the intent runtime, not strand the
  action in an opaque intermediate state

### Concurrent intents

The same account may receive overlapping requests from multiple terminals or
agents.
The stack should therefore model:

- conflict detection between pending plans
- bounded reservation or nonce-consumption semantics where needed
- owner-visible explanations when one terminal invalidates or supersedes
  another terminal's pending action

## Threat Model and Trust Boundaries

The 2046 architecture should explicitly identify the attack surfaces it is
being built to contain.

### Weak terminal threats

- stolen cards or terminal clones
- compromised kiosks
- replay of old terminal approvals
- terminal-triggered requests outside the allowed authority class

### Human-interface spoofing threats

- voice impersonation
- deceptive recovery prompts
- fake merchant presentation
- manipulated approval narration

### Provider and sponsor threats

- malicious or biased routing
- quote manipulation
- sponsor substitution
- hidden fee extraction
- censorship of preferred execution paths

### Agent-runtime threats

- compromised local OpenFox runtime
- unsafe plugin or adapter behavior
- privilege escalation between agent roles
- stale plan reuse after policy changes

### Contract and artifact threats

- artifact substitution
- ABI or manifest mismatch
- hidden high-risk effects
- unverifiable callback or settlement behavior

### Recovery and governance threats

- guardian collusion
- fraudulent recovery flow
- revocation not propagating fast enough
- weak evidence in disputes and audits

The key design objective is not to eliminate all trust,
but to minimize silent trust, bound authority, and preserve evidence.

## Privacy at Weak Terminals

The existence of public and privacy-tier asset paths means terminal design
cannot be neutral about privacy.

The architecture should assume:

- public or shared terminals are bad places to reveal private balances,
  counterparties, or memos
- voice and kiosk surfaces have additional observation risks such as
  bystanders, recording, and coerced narration
- private transfer policy cannot depend on the terminal learning full
  transaction detail

Practical consequences:

- privacy-tier actions should default to stricter terminal classes than public
  TOS actions
- weak terminals should request bounded privacy actions, not unrestricted
  private transfer authority
- policy checks for privacy-tier actions should rely on disclosed bounds,
  spend classes, or proof-carrying limits, not on full amount disclosure to
  the terminal surface
- public terminal UX should prefer redacted outputs and step-up approval for
  private actions

## Guardian and Recovery Model

Guardian and recovery semantics should not remain a placeholder phrase.

A guardian should be treated as a policy role, not as a specific UI surface.
The actual guardian may be:

- an on-chain address
- a threshold set of addresses
- a policy-recognized agent or service

The recovery model should include at minimum:

- explicit recovery initiation records
- time-lock and challenge windows
- owner alerting and observable evidence
- revocation propagation for lost terminals, session scopes, and delegated
  authority
- bounded post-recovery authority until normal control is re-established

Mitigations against guardian collusion or fraudulent recovery should include:

- threshold guardians instead of one guardian where possible
- delayed recovery activation
- owner-visible recovery receipts and proof references
- policy classes that separate everyday spend authority from recovery authority

## Schema Versioning and Migration

The shared objects in this document are guaranteed to evolve.
That evolution should be designed now rather than after the first drift.

Rules:

- every shared object should carry a schema identifier and version
- additive changes should be preferred over breaking field reuse
- breaking changes should use dual-read and dual-write migration windows where
  practical
- OpenFox, TOL, and GTOS should publish a compatibility matrix keyed by shared
  schema package version
- TOL artifact metadata versioning and shared-boundary versioning should be
  related but not collapsed into one number

The goal is not to freeze the first schema forever.
The goal is to keep evolution explicit, reviewable, and synchronized across the
three repositories.

## End-to-End Example: Card-Present Retail Payment

The following flow shows how the architecture should feel in practice.

1. A user taps a payment card at a merchant POS terminal.
2. The POS is treated as a weak terminal and sends a bounded payment intent.
3. OpenFox receives the request, identifies the merchant, terminal class,
   trust tier, and requested amount.
4. OpenFox compares available providers and sponsors and produces one or more
   executable plans.
5. The policy wallet checks merchant allowlist rules, spend caps,
   terminal-class permissions, and escalation thresholds.
6. TOL artifact metadata explains which contract or payment flow will execute,
   what effects are expected, and whether the operation remains inside a safe
   bounded policy class.
7. GTOS executes the sponsored settlement path and anchors the resulting
   receipt.
8. OpenFox stores the local intent journal, approval narrative,
   plan selection, and receipt linkage for later audit.

The same account should be able to perform this action without the POS terminal
becoming the root of trust.

## 12-Month Critical Path

The near-term execution spine should be compressed into four non-negotiable
deliverables.
The Shared Boundary Package is the hard gate.
It should have an explicit owner, release process, and hard deadline in the
actual program plan, because downstream phases otherwise drift by default.

### 1. Shared Boundary Package

Deliver one versioned schema package for:

- `IntentEnvelope`
- `PlanRecord`
- `ApprovalRecord`
- `ExecutionReceipt`
- terminal classes and trust tiers
- artifact identity and proof references
- agent role and account binding metadata

### 2. Policy Wallet MVP

Deliver one production-usable policy wallet substrate that supports at minimum:

- spend caps
- allowlists

### 3. One Real Multi-Terminal Demo

### 4. Auditable Intent-to-Receipt Path

Deliver one path where the same action can be inspected as:

- intent
- plan

- receipt

If the stack cannot prove one real multi-terminal path and one real
intent-to-receipt audit path, the architecture risks remaining only a
specification.

### Parallelization and Dependencies

The roadmap should not be interpreted as "only one repository moves at a
time."
It should be interpreted as:

- Phase 1 is the hard schema gate
- once Phase 1 is stable enough to release, OpenFox intent work and TOL
  metadata stabilization can run in parallel
- GTOS policy-wallet work can begin in parallel with TOL template work once the
  shared receipt and artifact boundary is stable enough
- multi-terminal product work can start early at the adapter-interface level,
  but terminal-specific authority classes should not ship ahead of policy
  wallet safeguards

The point is to parallelize implementation while keeping contract boundaries
serialized and explicit.

## Final Recommendation

The strategic architecture should be stated plainly:

> OpenFox is not "the frontend for GTOS."  
> OpenFox is the intent runtime and terminal operating layer.

> TOL is not "just a compiler behind GTOS."  
> TOL is the agent-native contract and policy language layer.

> GTOS is not "the backend chain behind OpenFox."  
> GTOS is the settlement, policy, and agent-infrastructure substrate.

This is the split that supports the 2046 world:

- users interact through many weak terminals
- authority is enforced through policy-bound accounts
- contract semantics are machine-readable before execution
- execution is sponsor-friendly and gas-abstracted
- agents are discoverable and comparable
- every meaningful action is auditable

If this split is designed now, the system can evolve from today's CLI, wallet,
discovery, gateway, paymaster, policy, and agent-native contract surfaces into
a true agent-native financial fabric without a foundational rewrite later.

The end state is a system where users and agents perform auditable financial
actions through any terminal, under policy constraints, expressed as intents,
without ever needing to know that a blockchain exists underneath.
