## CRITICAL

### C-1: Checkpoint vote gossip amplification (P2P)

**File**: `tos/handler.go` â€” `BroadcastCheckpointVote` (line 565), `RelayCheckpointVote` (line 575)

**Problem**: Both functions send to all connected peers with no deduplication or seen-set.
When a node receives a vote via `handleNewCheckpointVote`, it relays to all peers except
the sender. An attacker can send the same vote envelope repeatedly, and every receiving
node re-broadcasts it to all its peers, creating exponential gossip amplification.

**Impact**: Network-wide bandwidth exhaustion. A single malicious peer can flood the
entire validator network with minimal effort.

**Fix**: Add an LRU seen-set keyed on `keccak256(RLP(CheckpointVoteEnvelope))`. Skip
relay if the vote hash is already in the set. Add per-peer rate limiting on
`NewCheckpointVoteMsg`.

**Verified**: CONFIRMED â€” code reviewed; no seen-set, no LRU, no dedup exists.
Compare with `BroadcastBlock` which uses `sqrt(N)` fan-out; votes have no such limit.

**Status**: FIXED â€” `99c80c1` added LRU seen-set (`seenVotes *lru.Cache`, capacity 1024)
keyed on `keccak256(vote.Hash || signer)`. `BroadcastCheckpointVote` marks seen before
sending; `RelayCheckpointVote` skips already-seen votes.

### C-2: Checkpoint votes relayed before signature verification (P2P)

**File**: `tos/protocols/tos/handler.go` â€” `handleNewCheckpointVote` (lines 222-229)

**Problem**: The handler decodes the vote and passes it directly to `backend.Handle`
without verifying the ed25519 signature. If the backend calls `RelayCheckpointVote`
before validation (which is the natural flow), invalid votes propagate network-wide.

**Impact**: Amplifies invalid/garbage messages across the network, wasting bandwidth
and CPU on all nodes.

**Fix**: Verify the ed25519 signature before calling `RelayCheckpointVote`. Only relay
votes that pass cryptographic validation.

**Verified**: CONFIRMED â€” `handler_tos.go:66-67` calls `CheckpointVoteHandler` then
`RelayCheckpointVote` unconditionally. `HandleIncomingVote` (dpos.go:354) verifies
signatures but returns no error/bool; the relay on line 67 runs regardless of result.

**Status**: FIXED â€” `99c80c1` changed `HandleIncomingVote` to return `bool`;
`handler_tos.go` only calls `RelayCheckpointVote` when handler returns `true`
(i.e., after ed25519 signature verification and vote pool admission pass).

### H-4: Per-tx gas pool vs block gas pool semantic difference

**File**: `core/parallel/executor.go`

**Problem**: In geth, the block gas pool is decremented by `msg.Gas()` upfront (via
`buyGas`), temporarily reserving the full declared gas, then the unused portion is
refunded. In GTOS parallel execution, each tx gets its own per-tx gas pool seeded with
`msg.Gas()`, and only `result.UsedGas` is deducted from the block gas pool during merge.

**Verified**: CONFIRMED as divergence, but NOT a bug. The gate check on `executor.go:180`
(`msgs[txIdx].Gas() > gp.Gas()`) ensures the declared gas fits in the block. The net
deduction (`usedGas`) is identical to geth's reserve-then-refund model. Block validation
produces consensus-equivalent results.

**Status**: BY DESIGN â€” no fix needed

## MEDIUM

### M-1: AA txHash omits gas parameters

**File**: `core/state_transition.go` â€” AA validation (lines 300-305)

**Problem**: The txHash passed to `validate(bytes32, bytes)` is
`keccak256(chainID || from || to || nonce || value || data)`. It does not include
`gasLimit` or `gasPrice`. An attacker can submit the same logical transaction with
different gas parameters and get the same AA txHash.

**Impact**: If a validate() function uses txHash for replay protection (e.g., marking
hashes as "seen"), varying gas params bypasses the protection, allowing the same logical
operation to execute multiple times.

**Fix**: Include `gasLimit` in the AA txHash computation, or document this as a known
limitation for AA contract developers.

**Verified**: CONFIRMED â€” lines 292-304 build txHashInput from only chainID, from, to,
nonce, value, data. No gas fields.

**Status**: FIXED â€” `99c80c1` added `gasBuf` (msg.Gas()) to txHashInput between
nonce and value, so AA txHash now includes gasLimit.

### M-3: WriteBufStateDB Suicide/HasSuicided are no-ops

**File**: `core/parallel/writebuf.go` â€” line 220

**Problem**: `Suicide()` returns false and does nothing. `HasSuicided()` returns false.
If any LVM execution path triggers SELFDESTRUCT semantics, the account will not be
marked for deletion in the overlay, and the merge step will not propagate destruction.

**Impact**: Accounts that should be destroyed persist with stale state. Potential
fund-locking or ghost-account issues.

**Fix**: If GTOS intentionally does not support SELFDESTRUCT, add a panic or error
return so any accidental invocation is caught immediately. If it should be supported,
implement proper overlay tracking.

**Verified**: CONFIRMED â€” `Suicide` returns false unconditionally (line 220),
`HasSuicided` returns false unconditionally (line 221). Merge has no suicide logic.

**Status**: FIXED â€” `99c80c1` changed `Suicide` and `HasSuicided` to
`panic("WriteBufStateDB: ... not supported")`, catching any accidental invocation.

### M-4: WriteBufStateDB refund tracking is a no-op

**File**: `core/parallel/writebuf.go`

**Problem**: `AddRefund()`, `SubRefund()`, `GetRefund()` are all no-ops in the write
buffer overlay. Gas refund mechanics (used by SSTORE gas refunds) are silently dropped.

**Impact**: Transactions that would receive gas refunds in serial execution do not
receive them in parallel execution, causing higher gas consumption.

**Fix**: If GTOS has no SSTORE gas refund model, document this explicitly. Otherwise,
implement refund tracking in the overlay.

**Verified**: CONFIRMED â€” `AddRefund`, `SubRefund` discard arguments; `GetRefund`
returns 0. `refundGas()` in state_transition.go:717 calls `GetRefund()` which always
returns 0 through WriteBufStateDB.

**Status**: FIXED â€” `99c80c1` added explicit comments documenting that GTOS LVM
has no SSTORE gas refund model; refund no-ops are intentional by design.

### M-6: txSenderHint falls back to unverified From field

**File**: `core/types/transaction.go` â€” `txSenderHint` (lines 503-511)

**Problem**: For non-secp256k1 transactions, `Sender()` always fails, so
`txSenderHint()` returns the unverified self-declared `From` field. This function is
used in `NewTransactionsByPriceAndNonce` for transaction ordering during block building.

**Impact**: A non-secp256k1 tx with a spoofed `From` that somehow bypassed pool
validation could be ordered under the wrong account. Does not affect fund security
(tx_pool uses `ResolveSender` for verification), but could affect block building
fairness.

**Fix**: Have `txSenderHint` use the sender cache populated by `ResolveSender` instead
of falling back to the raw `From` field.

**Verified**: CONFIRMED â€” `accessListSigner.Sender()` returns `ErrTxTypeNotSupported`
for non-secp256k1. Fallback to `tx.SignerFrom()` returns unverified `From` field.

**Status**: FIXED â€” `99c80c1` replaced `tx.SignerFrom()` fallback with
`tx.from.Load()` sigCache check, using only cryptographically verified senders.
