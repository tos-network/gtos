# GTOS vs geth Deep Audit — Round 3 (2026-03-08)

Baseline: go-ethereum v1.10.25 (~/geth) vs GTOS (~/gtos)

Scope: Six subsystems not covered in Rounds 1–2: checkpoint finality, account signer,
core types, sysaction handlers, UNO confidential transactions, worker/miner pipeline.

Each issue independently verified against source code. Prior findings from
Round 1 (BUGS-20260308.md) and Round 2 (BUGS-20260308-R2.md) are excluded.

---

## Severity Levels

- **CRITICAL**: Consensus-breaking or chain-halting bug. Must fix before production.
- **HIGH**: Active exploit vector or DoS. Should fix before production.
- **MEDIUM**: Correctness gap, compatibility break, or defense-in-depth issue.
- **LOW**: Minor issue, informational, or cosmetic.

## CRITICAL

### CR-1: Empty block sealed before scheduled tasks — wrong state root

**File**: `miner/worker.go` — `commitWork()` (lines 1149–1172)

**Problem**: `commitWork` calls `commit(work.copy(), nil, false, start)` at line 1150
BEFORE `RunScheduledTasks` executes at line 1158. The commit method calls
`FinalizeAndAssemble` → `Finalize` which computes a state root reflecting only block
reward (no scheduled tasks, no user txs). This block is sent to `Seal()`. If the seal
completes before the second `commit()` at line 1172 interrupts it, this empty block
with an incorrect state root is broadcast.

Meanwhile, every validating node runs `Process()` which executes `RunScheduledTasks`
BEFORE user transactions (state_processor.go:88), producing a different state root.

**Impact**: Consensus split. The miner forks away from the network whenever scheduled
tasks exist for a block.

**Verified**: CONFIRMED — line 1150 commits before line 1158 runs tasks. `Finalize`
computes state root without task execution.

**Status**: FIXED — moved `RunScheduledTasks` before the empty-block pre-seal commit.

---

### CR-2: Timestamp unit mismatch — seconds vs milliseconds in newWorkLoop

**File**: `miner/worker.go` — `newWorkLoop()` (lines 473, 478)

**Problem**: `timestamp = time.Now().Unix()` produces seconds. DPoS uses millisecond
timestamps (`time.Now().UnixMilli()` in dpos.go:1506). `parent.Time()` stores
milliseconds. The sanity check `parent.Time() >= timestamp` is ALWAYS true (ms > s),
causing `timestamp = parent.Time() + 1` on every block. While `Prepare()` overwrites
the timestamp, the `getSealingBlock` path with `forceTime: true` is broken — always
returns error unless caller passes millisecond timestamps.

**Impact**: `getSealingBlock` API broken for external block builders. Pre-Prepare
sanity check is dead code.

**Verified**: CONFIRMED — lines 473, 478: `time.Now().Unix()`. DPoS Prepare at
dpos.go:1506: `time.Now().UnixMilli()`.

**Status**: FIXED — changed both to `time.Now().UnixMilli()`.

## HIGH

### H-1: Deferred tasks re-enqueued regardless of cancellation/completion status

**File**: `task/processor.go` — lines 138–143

**Problem**: In the overflow-handling loop, `EnqueueTask(db, blockNum+1, tid)` at
line 143 executes OUTSIDE the `if rec.Status == TaskPending` guard at line 139.
Cancelled/done/expired tasks get re-enqueued to the next block indefinitely.

```go
for _, tid := range deferred {
    if rec, ok := ReadTask(db, tid); ok && rec.Status == TaskPending {
        rec.TargetBlock = blockNum + 1
        WriteTask(db, tid, rec)
    }
    EnqueueTask(db, blockNum+1, tid)  // <-- ALWAYS runs
}
```

**Impact**: Ghost task IDs accumulate in the queue, wasting state reads every block.
Cancelled tasks become zombies that can never be cleaned up.

**Verified**: CONFIRMED — `EnqueueTask` at line 143 is outside the `if` guard.

**Status**: FIXED — moved `EnqueueTask` inside the `if rec.Status == TaskPending` guard.

### H-2: Pending vote queue unbounded — memory DoS on syncing nodes

**File**: `consensus/dpos/finality_pool.go` — `AddPending()` (line 111)
**File**: `consensus/dpos/dpos.go` — `HandleIncomingVote()` (lines 386–402)

**Problem**: `AddPending` appends votes with no per-number cap and no dedup.
`HandleIncomingVote` has no upper-bound check on vote number. `Prune()` is
unreachable on the pending-return code paths (lines 393, 402 return before line 440).

**Impact**: A single malicious peer can OOM a syncing node by flooding checkpoint
votes for future numbers or duplicating votes for recent ones.

**Verified**: CONFIRMED — `AddPending` unconditionally appends; no dedup, no cap.
No upper-bound check on vote number. `Prune` unreachable on pending paths.

**Status**: FIXED — added per-signer dedup and cap=50 in `AddPending`; upper-bound
check and `Prune` on pending paths in `HandleIncomingVote`.

### H-3: Unshield/transfer `payload.To` not restricted from system addresses

**File**: `core/state_transition.go` — lines 642, 692–734
**File**: `core/uno/codec.go` — lines 147, 180–199

**Problem**: Unshield credits native TOS to `payload.To` via `AddBalance`. The only
check is `To != zero address`. No check prevents `To` from being system addresses
(`SystemActionAddress`, `PrivacyRouterAddress`, `CheckpointSlashIndicatorAddress`).

**Impact**: Funds unshielded to system addresses are effectively burned or could
interfere with system action gas accounting.

**Verified**: CONFIRMED — `DecodeUnshieldPayload` checks zero address only.
`DecodeTransferPayload` same pattern.

**Status**: BY DESIGN — only zero address is restricted. Sending to system addresses
is the user's responsibility; no additional restrictions needed.

### H-4: Transfer range proof optional — no mandatory non-negativity check

**File**: `core/uno/verify.go` — lines 109–111
**File**: `core/uno/prover.go` — lines 87–146

**Problem**: Transfer proof bundles accept two sizes: 360 bytes (no range proof) or
1032 bytes (with range proof). At verify.go:109, if the range proof portion is empty,
verification returns `nil` (success) without any range check. The prover at
prover.go:137–139 never generates a range proof.

**Impact**: If the balance + CT-validity proofs do not implicitly constrain the
committed amount to [0, 2^64), an attacker could create or destroy confidential value.

**Verified**: CONFIRMED — `VerifyTransferProofBundleWithContext` short-circuits when
`len(parts.rangeProof) == 0`. `BuildTransferPayloadProof` never includes range proof.

**Status**: FIXED — Range proofs are now mandatory for transfers and match the intended confidential-transfer design.
Transfer proof bundle is exactly 1032 bytes (CT 160 + Balance 200 + RangeProof 672).
Verifier rejects bundles without range proof. Prover generates Bulletproofs range proof
via `at_rangeproofs_prove_single64()` C implementation. All tests pass.

## MEDIUM

### M-1: ErrTaskLogsNotAllowed aborts entire block — liveness griefing

**File**: `task/processor.go` — lines 74–77

**Problem**: If a scheduled task callback emits a log, `ProcessDueTasks` returns
`ErrTaskLogsNotAllowed` as a fatal error. This aborts the entire block in both
`commitWork` and `Process()`. The task remains Pending and fires again next attempt,
creating a permanent block production halt.

**Impact**: Permanent liveness failure. Task registration is permissionless, so this
is a griefing vector.

**Verified**: CONFIRMED — processor.go:76 returns fatal error. Task not marked
Done/Expired before return.

**Status**: FIXED — task marked `TaskExpired`, processing continues to remaining tasks.

### M-2: `targetBlock` uint64 overflow in task scheduling

**File**: `task/handler.go` — line 100
**File**: `task/processor.go` — line 111

**Problem**: `targetBlock := blockNum + p.DelayBlocks` and `nextBlock := blockNum +
rec.IntervalBlocks` can overflow uint64. `IntervalBlocks` has only a minimum check
(>= 10), no maximum. A recurring task with `IntervalBlocks = MaxUint64 - blockNum + 1`
wraps to 0, enqueuing to an already-processed block.

**Impact**: Permanently stranded task with locked deposit.

**Verified**: CONFIRMED — no overflow check or max interval bound.

**Status**: FIXED — added `TaskMaxHorizonBlocks` max for `IntervalBlocks`; overflow
check sets task to `TaskExpired`.

### M-3: Slashing evidence accepted for non-validators

**Problem**: `Execute` validates evidence cryptographically but does NOT check that

**Impact**: Spurious evidence pollutes on-chain records; wastes storage.

**Verified**: CONFIRMED — no validator set membership check in `Execute`.

**Status**: FIXED — added `validator.ReadValidatorStatus` check; rejects non-validators.

**File**: `agent/handler.go` — lines 95–104
**File**: `agent/state.go` — `WriteMetadata()` (line 142–145)

**Problem**: `WriteMetadata` stores only the first 32 bytes of the URI via

### M-5: Capability grant/revoke accept unregistered bit indices

**File**: `capability/handler.go` — lines 63–87

**Problem**: `handleGrant`/`handleRevoke` accept any `bit` (uint8, 0–255) without
checking whether the index has been registered via `RegisterCapabilityName`. If an
unregistered bit is pre-granted and later formally registered, the address silently
gains the new capability without authorization.

**Impact**: Privilege escalation vector.

**Verified**: CONFIRMED — no `readBitCount` check in grant/revoke handlers.

**Status**: FIXED — added `p.Bit >= readBitCount` check; returns `ErrCapabilityBitUnregistered`.

### M-9: secp256r1 signing functions do not normalize S to low-S

**File**: `accountsigner/crypto.go` — `SignSecp256r1Hash()` (line 601),
`EncodeSecp256r1Signature()` (line 566)
**File**: `core/types/transaction_signing.go` — `signSecp256r1Hash()` (line 211)

**Problem**: R2 M-4 added low-S enforcement in `VerifyRawSignature`. But signing
functions don't normalize S. Go's `ecdsa.Sign` produces low-S in practice, but
`EncodeSecp256r1ASN1Signature` (for HSMs/WebAuthn) accepts externally-produced DER
signatures which commonly have high-S (~50% probability).

**Impact**: External P-256 signers (HSMs, WebAuthn, FIDO2) have transactions rejected
~50% of the time.

**Verified**: CONFIRMED — no S normalization in any signing function.

**Status**: FIXED — added low-S normalization in both `EncodeSecp256r1Signature`
(crypto.go) and `encodeSecp256r1Signature` (transaction_signing.go).
